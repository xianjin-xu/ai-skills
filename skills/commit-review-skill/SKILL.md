# Commit Review Skill

## 功能概述

对本地 Git 仓库的 commit 进行代码审查，支持：
- 审查当前分支的最新 commit
- 审查指定 commit hash 的变更
- 自动对比与上游分支的差异
- 输出结构化的 Review 报告

## 使用场景

- 提交前自我检查代码质量
- Code Review 前的预审
- 检查合并后的代码质量

## 执行流程

### 步骤 1：获取 Commit 信息

```bash
# 获取当前分支最新 commit
git log -1 --format="%H|%an|%ae|%ai|%s"

# 或指定 commit
git show <commit-hash> --stat

# 查看变更的文件列表
git show <commit-hash> --name-only --format=""
```

### 步骤 2：获取完整 Diff

```bash
# 获取完整 diff
git show <commit-hash> -p

# 如果需要对比分支差异
git diff <base-branch>..<commit-hash> -p
```

### 步骤 3：验证编译

```bash
# .NET/C# 项目
dotnet build

# Java 项目
mvn compile

# Node.js 项目
npm run build
```

### 步骤 4：AI Code Review

以 **资深后端架构师** 身份进行深度 Code Review。

#### 审查维度

| 维度 | 是否严重 | 审查要点 | 问题示例 |
|------|----------|----------|----------|
| **逻辑正确性** | ✅ 是 | 边界条件、异常分支、业务流程 | 逻辑错误？业务流程不正确？ |
| **资源管理** | ✅ 是 | 连接关闭、内存释放、异常时资源释放 | 数据库连接泄露？文件句柄未关闭？ |
| **并发安全** | ✅ 是 | 共享变量、静态字段、线程安全 | 多线程访问是否有锁保护？ |
| **安全风险** | ✅ 是 | 敏感信息、注入漏洞、参数校验 | 密码明文？SQL 注入？XSS？ |
| **代码质量** | ❌ 否 | 重复代码、方法过长 | 代码重复、方法过长（归为改进建议） |
| **命名规范** | ❌ 否 | 变量命名、方法命名 | 命名不规范（归为改进建议） |
| **空值检查** | ⚠️ 视情况 | null 检查、默认值处理 | 已有防御代码保护的不算严重问题 |
| **Magic String** | ❌ 否 | 硬编码字符串 | 硬编码敏感信息才算严重问题 |
| **Git 规范** | ❌ 否 | commit message、提交粒度 | 归为改进建议 |

#### 关键审查问题

对每个修改的文件，逐条检查：
- 这段代码在什么场景下会出错？
- 如果输入是 null/空/边界值会怎样？
- 异常情况下是否正确处理？
- 是否有资源泄露风险？
- 变量命名是否清晰易懂？
- 是否符合项目代码规范？

### 步骤 5：评分与合并建议

**严重问题（Must Fix）标准**：
- 会导致功能错误、数据丢失、安全漏洞、运行时崩溃
- 逻辑错误，业务流程不正确
- 资源泄露（数据库连接、文件句柄等）
- SQL 注入、XSS 等安全漏洞
- 硬编码密码/密钥/秘钥等敏感信息泄露

**不构成严重问题的情况**：
- 已有防御性代码保护的空值检查（即使写法不够优雅）
- 代码风格不一致（除非影响功能）
- 命名不规范
- 未使用的代码（需要具体分析）
- Magic String（除非是敏感信息）

| 问题级别 | 数量 | 合并建议 |
|----------|------|----------|
| 严重问题 | ≥1 | ❌ 需要大幅修改 |
| 严重问题 | 0 且 改进建议 > 10 | ⚠️ 建议修改后合并 |
| 无严重问题 | ≤10 个改进建议 | ✅ 可以合并 |
| 无严重问题 | 0 个问题 | ✅ 建议合并 |

## 输出格式

```markdown
# Commit Review 报告

**Commit** `<hash>` | **分支: <branch-name>**

## 编译状态
✅ 编译通过 / ❌ 编译失败

## 变更概述
- Commit 信息：<message>
- 文件变更：<n> 个文件 (+<add> / -<del> 行)

## 审查结果

### 🔴 严重问题（Must Fix - 必须修复，阻止合并）
- **[问题标题]** (<文件>:<行号>)
  - 问题描述：<具体描述>
  - 影响：<会导致什么问题>
  - 建议：<修复方案>
  - 判定依据：⚠️ 需确认是真正的功能性错误，而非代码风格问题

### 🟡 改进建议（Should Fix - 推荐处理）
- **[改进标题]** (<文件>:<行号>)
  - 问题描述：<具体描述>
  - 建议：<改进方案>

### 🟢 代码亮点
- <亮点描述>

## Git 规范检查
- Commit Message：✅ 规范 / ⚠️ 需要改进
- 提交粒度：✅ 合理 / ⚠️ 建议拆分
- 变更范围：✅ 适中 / ⚠️ 变更过大

## 总结
- 严重问题数量：<n> 个（功能性错误、崩溃、数据丢失、安全漏洞等）
- 改进建议数量：<n> 个（代码风格、命名、规范性等）
- 合并建议：⚠️ 建议修改后合并 / ✅ 可以合并 / ❌ 需要大幅修改

**重要提醒**：
- 严格区分"严重问题"和"改进建议"，避免将代码风格问题误判为严重问题
- 已有防御性代码保护的空值检查不是严重问题
- 如果代码能编译通过且功能正常，应倾向于给出"✅ 可以合并"
```

## 常用命令参考

```bash
# 查看当前分支最新 commit
git log -1

# 查看指定 commit 的变更统计
git show <hash> --stat

# 查看完整 diff
git show <hash> -p

# 对比两个 commit 的差异
git diff <old-hash>..<new-hash>

# 查看某个文件在 commit 中的变更
git show <hash> -- <file-path>

# 撤销最新 commit（保留变更）
git reset --soft HEAD~1

# 撤销最新 commit（不保留变更）
git reset --hard HEAD~1
```
